Self-improvement loop (CLAUDE.md §9) for task 017. Full stage record:

## What happened

The fix was implemented first (founder asked for it "right now"), then the full
workflow ran retroactively: product → plan → architect (ADR-013) → executor
(tests + tag-escape hardening) → review / QA / security in parallel → fix round →
docs → this PR. All gates passed: review APPROVE-WITH-NITS, QA Pass (AC-1..8),
security PASS-WITH-NOTES; the security should-fix (case/whitespace-tolerant tag
escaping) was applied before merge.

## What slowed development down

1. **Shared working tree with a concurrently active session.** Mid-workflow the
   branch switched (`security/harden-web-api` → `feat/background-chat-runs`),
   commit `cd6286c` swept in one of our uncommitted doc edits, and unrelated
   web/analytics changes accumulated alongside ours. The PR had to be assembled
   by porting files/hunks into a clean worktree off `main`. Second occurrence of
   this pattern (016 shipped from a mixed tree too). **Third occurrence → make a
   skill** (`skills/` candidate: "dedicated-PR from a shared working tree":
   worktree off main, port only task-scoped hunks, verify, PR).
2. **Stale architecture claim cost a placement correction.** The design record
   said the `apps/api/tests/` tier didn't exist on the branch; it did
   (`pytest.ini` `testpaths = tests`), so the new test file was first created
   root-level where bare `pytest` never collects it. Caught at review. Lesson
   folded into the task file; the tests now live in `tests/unit/`.
3. **Retroactive workflow is workable but lossier.** Product/planner/architect
   had to reverse-engineer intent from a finished diff. The stages still caught
   real issues (block-escape hardening came from the architect's risk list), but
   the order inverted the cheap-feedback property the workflow exists for.
