## What slowed development

- **Session usage limit** killed the executor mid-stage; resumable via
  SendMessage with context intact, but it forced the QA-stage compromise.
  Lesson: for large tasks, the implementation stage is the budget hog — brief
  it to read economically (it re-read files it knew from the prior task).
- **Stale venv vs merged main**: PR #18 (PostHog) added a dependency the main
  checkout's venv predated; 2 pre-existing test failures until `pip install
  posthog` — not a 020 defect, but it cost a debug loop. Candidate check:
  `pip install -r requirements.txt` before any test run on a fresh-merged base.

## Knowledge that was missing (now captured)

- `get_memory_manager()` ignored `EMBEDDING_PROVIDER` (raw `OllamaEmbeddings`
  hardcoding `localhost:11434`) — memory embeddings were silently broken
  off-laptop. Fixed in 020 via `get_default_embedder`; recorded in ADR-014.
- The identity mess (Clerk id vs DB uuid vs planner string ids) again cost
  analysis time; ADR-014 documents the Clerk-id decision for `memory_pages`
  and flags the research-engine uuid mismatch as backlog.

## Repeated activity → skill/workflow candidates

  (017, 020). One more and it should become a `skills/` or `meta/` runbook:
  worktree setup, venv reuse, stage briefs, gate fix-round, M6 closing moves.
