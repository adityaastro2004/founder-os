## 1. Executive Summary

Founder OS is a **multi-agent AI backend** built with FastAPI, PostgreSQL+pgvector, Redis, and Celery. The codebase is significantly more complete than the README suggests. Several features listed as "near-term roadmap" are already built, and several major systems exist that the README doesn't mention at all.

**Key stats:**
- **~50+ API endpoints** (README documents only 5)
- **~26 database tables** (README says "24")
- **7 concrete agents** + Orchestrator
- **6 LLM provider classes** (README mentions 3)
- **12 built-in tools** (most are placeholder stubs)
- **~15,000+ lines of Python** in `apps/api/`

```
AgentRegistry.get(name, user_id)

  → loads DB config (agents table) + user overlay (user_agent_configs table)
  → creates: LLMProvider, ToolRegistry, AgentMemory (4-layer), AgentRouter, EventBus, ApprovalGate
```

## 8. Temporal Memory System (memory_pages)

> File: `memory/manager.py` (1,069 lines) — this is separate from the agent-level memory and supports the `/api/memory/*` endpoints.

### Composite Scoring Formula
```
score = (semantic_sim × 0.35)
      + (temporal_relevance × 0.30)
      + (importance × 0.20)
      + (access_boost × 0.15)

where temporal_relevance = importance × exp(-decay_rate × days_since_occurred)
```

### Features
- **Store/Recall/Review lifecycle** with both sync and async variants
- **Spaced-repetition** review scheduling (Leitner-like intervals: 1, 3, 7, 14, 30, 60, 120, 240 days)
- **Chapter-based browsing** (product, hiring, fundraising, marketing, ops, growth, support)
- **Entity tracking** (people, companies, tools referenced — JSONB)
- **Typed links** between memories: `related`, `caused_by`, `led_to`, `contradicts`, `updates`, `supersedes`, `part_of`
- **Pin/unpin** — pinned items never decay
- **Soft delete**
- **Auto-embedding** via configured embedding provider
- **PostgreSQL function** `memory_temporal_score()` for server-side scoring
- **LLM context formatting** — `format_for_llm()` outputs XML-structured memories

### Database Tables
- `memory_pages` — ~25 columns including vector(1536) embedding, JSONB entities, temporal metadata
- `memory_links` — source_id, target_id, link_type, strength (unique constraint on triple)
- 10 indexes: temporal, chapter, review-due, importance, type, vector (ivfflat), full-text (GIN), tags (GIN), entities (GIN jsonb_path_ops)

## 9. Retrieval / RAG Subsystem

> Directory: `app/retrieval/` (5 files, ~1,500 lines)

| File | Class | Purpose |
|------|-------|---------|
| `embeddings.py` | `OllamaEmbeddings`, `OpenAIEmbeddings`, `CachedEmbeddingProvider` | Multi-backend embedding with Redis cache (7-day TTL, content-hash keys). Pads/truncates all vectors to 1536d. |
| `vector_store.py` | `VectorStore` | pgvector CRUD: upsert, bulk_upsert, semantic search (cosine), fulltext search (tsvector), hybrid search (RRF fusion), delete, count |
| `chunker.py` | `TextChunker` | Token-aware recursive chunking (tiktoken cl100k_base). Splits at paragraph → sentence → comma → space boundaries. Configurable chunk_size (512), overlap (50), min_chunk_size (30). |
| `ingester.py` | `Ingester` | Pipeline: content → chunk → batch-embed → store. Supports text, URL (HTML extraction), JSON, batch. |
| `retriever.py` | `ContextRetriever` | High-level interface: search (hybrid/semantic/fulltext), `get_context()` for XML-tagged LLM injection, usage tracking, stats |

### Hybrid Search (RRF — Reciprocal Rank Fusion)
Runs cosine similarity and full-text search in parallel, then fuses results using:
```
RRF_score = 1/(k + rank_semantic) + 1/(k + rank_fulltext)
```

## 10. Weekly Planner & Google Calendar

### Planner Routes (`planner_routes.py` — 1,333 lines)

**Smart NL Endpoint** (`POST /api/planner/prompt`):
```
User prompt
    → Memory recall (last N relevant memories)
    → Google Calendar context (upcoming events)
    → LLM intent extraction (events to create, context updates, replan?)
    → Execute: create GCal events + update user context + optional full replan
    → Store interaction as memory page
    → Return response
```

**Plan Generation** (`POST /api/planner/generate`):
```
User context (business profile + goals + metrics + memory)
    → LLM generates JSON directly (WeeklyPlan schema)
    → Validation + fallback repair (re-ask LLM if JSON is invalid)
    → ensure_daily_schedule() (auto-distribute priorities if empty)
    → Push all tasks to Google Calendar
    → Store plan in plan_history
    → Store plan summary as memory page
```

### Planner Models (`planner_models.py` — 476 lines)
- `WeeklyPlan`: root model with top_priorities, daily_schedule, delegations, risks, success_criteria
- `Priority`: rank, title, rationale, ICE score, owner_agent
- `DaySchedule`: day, date, tasks[], notes
- `PlanTask`: title, description, owner_agent, priority, est_hours, start_time, end_time, ICE score, tags
- `ICEScore`: Impact × Confidence × Ease (each 1-10), auto-computed total
- `parse_plan_to_model()`: LLM post-processor (markdown → JSON extraction)
- `plan_to_ical()`: generates .ics calendar data

### Google Calendar Integration (`calendar_integration.py` — 599 lines)
- OAuth2: `get_auth_url()`, `exchange_code_for_tokens()`, `refresh_access_token()`
- Token management: PostgreSQL-backed via user_store, auto-refresh on expiry
- `push_plan_to_gcal()`: pushes entire WeeklyPlan (all daily tasks → individual events)
- `create_single_event()`, `create_all_day_event()`, `update_event()`, `delete_event()`
- `list_events()`, `get_event()` (for calendar browsing)

### Scheduler (`scheduler.py` — 351 lines)
- APScheduler `AsyncIOScheduler` with `CronTrigger`
- `automated_planner_job`: runs every Monday 08:00 IST (production) / 30s after startup (dev)
- Iterates all users with connected GCal, generates personalized plans, pushes events
- Updates user stats (plan_count, last_plan_at, last_plan_events)

## 11. Approval System (Human-in-the-Loop)

> File: `agents/approval.py` (537 lines)

### Risk Levels
| Level | Behavior | Example Tools |
|-------|----------|---------------|
| **LOW** | Check user preference; default: ask | search_knowledge, web_search, get_business_metrics, list_tasks |
| **MEDIUM** | Check user preference; default: ask | create_task, save_draft, delegate_task |
| **HIGH** | **ALWAYS require approval**, no bypass ever | push_pr, send_email, deploy, make_payment, post_to_social_media (31 tools classified) |

### User Preferences (Redis-backed)
- `always_allow` — auto-approve (blocked for HIGH-RISK tools)
- `always_deny` — auto-reject
- `ask` — default, creates pending approval

### Flow
```
ExecutionEngine → ApprovalGate.check(tool_name, arguments)
    → classify_tool_risk()
    → HIGH → always create PendingApproval
    → LOW/MEDIUM → check ApprovalPreferences
        → always_allow → auto-approve
        → always_deny → auto-reject
        → ask → create PendingApproval in Redis (1h TTL)
    → User approves/rejects via API
    → Tool executes (or not)
```

## 12. Background Task Queue (Celery)

### Celery App (`celery_app.py`)
- Broker + backend: Redis
- 3 queues: `default`, `agents`, `orchestrator`
- JSON serialisation only
- `task_acks_late=True`, `worker_prefetch_multiplier=1`
- Soft limit: 5 min, hard limit: 6 min
- 4 workers (I/O-bound)

### Tasks (`tasks/agent_tasks.py` — 478 lines)
| Task | Queue | Retries | Backoff |
|------|-------|---------|---------|
| `run_agent_task` | agents | 3 | 10s → 40s → 160s (×4) |
| `run_orchestration_task` | orchestrator | 3 | 15s → 60s → 240s (×4) |

- Each task creates its own async DB session + Redis connection
- Stores status in Redis (`founder_os:task:{task_id}`) with 24h TTL
- Maintains per-user task index (sorted set, last 100 tasks)
- Transient error detection: timeout, rate limit, 429/502/503/504, connection errors

### Status Service (`tasks/status.py`)
- `TaskStatusService`: reads Redis status, falls back to Celery's `AsyncResult` for PENDING tasks
- `cancel_task()`: Celery `revoke(terminate=True, signal=SIGTERM)` + Redis status update

- `pgvector/pgvector:pg16` with healthcheck, volume `postgres_data`

## 16. What's Implemented vs. Stubbed

### Fully Implemented
- Agent orchestration (Stripe Minions pattern) with delegation trace
- 6 LLM providers with auto-fallback (Gemini 3-tier)
- Tool protocol with MCP-compatible interface
- MCP adapter (stdio + SSE clients)
- A2A agent routing with capability scoring
- Execution engine (agentic loop, parallel tool calls, approval gates)
- 4-layer agent memory (conversation, working, shared, long-term)
- Temporal memory system (composite scoring, spaced repetition, chapters, entities, links)
- Full retrieval pipeline (embed → chunk → ingest → hybrid search → context formatting)
- Embedding providers (Ollama + OpenAI) with Redis cache
- Approval system (risk classification, preferences, pending queue)
- Google Calendar integration (OAuth, CRUD, plan push)
- Weekly planner (NL prompt, LLM plan generation, calendar sync, memory storage)
- Background task queue (Celery, 3 queues, retry with backoff, status polling)
- Auth (Clerk JWT with JWKS)
- All 67 API endpoints

### Placeholder / Stub
- **Most built-in tools** — 9 of 12 tools return mock/placeholder data (search_knowledge, web_search, create_task, update_task_status, save_draft, get_business_metrics, list_tasks, get_integrations, store_working_memory). Comments say "wired to DB at runtime" but the wiring is not implemented in the registry.
- **External integrations** — Only Google Calendar is implemented. No Stripe, GitHub, Slack, Notion, Linear connectors.
- **MCP server config** — Client code exists but no way to configure/connect servers via config or API.
- **Frontend** — `apps/web/` and `apps/docs/` are default Next.js scaffolds (no custom UI).
- **Workflow persistence** — `workflow_templates`, `workflows`, `workflow_executions` tables exist in schema but no routes or logic use them.
- **Learning loop** — `task_feedback`, `learning_insights` tables exist but no routes or logic populate/consume them.
- **Streaming** — No SSE/WebSocket streaming from agents.
- **Multi-user teams** — Auth supports individual users; no team/org features beyond Clerk org_id claim passthrough.
